# AeroFlow Domain Model

## Назначение документа

Этот документ описывает доменную модель AeroFlow и правила её развития.

Он дополняет `README.md` и `architecture.md`:

* `architecture.md` определяет границы сервисов и взаимодействие между ними;
* `domain-model.md` определяет бизнес-понятия, агрегаты, инварианты и события.

Документ отражает как уже реализованную модель, так и целевую модель системы.
Нереализованные части явно отмечены как планируемые.

---

# Подход к DDD

AeroFlow использует pragmatic DDD.

DDD применяется там, где есть:

* жизненный цикл;
* бизнес-инварианты;
* переходы состояния;
* конкурентные изменения;
* значимые доменные события;
* правила, которые нельзя надёжно выразить только в controller или CRUD-сервисе.

Простой CRUD допустим для справочных данных, если у них нет собственного
поведения и сложных правил.

## Основные правила

1. Бизнес-решения принимаются в доменной модели или доменном сервисе.
2. Application layer координирует use case, транзакцию, репозитории и сообщения.
3. Infrastructure layer реализует хранение, транспорт, безопасность и интеграции.
4. API layer преобразует HTTP-запросы в команды и результаты в HTTP-ответы.
5. Агрегат является границей согласованности и изменяется через публичное поведение.
6. Между агрегатами предпочтительно хранить идентификаторы, а не объектные графы.
7. Репозиторий создаётся для aggregate root или отдельной транзакционной границы.
8. Domain event описывает уже произошедший значимый факт.
9. Integration command сообщает другому bounded context, что необходимо сделать.
10. Value Object вводится только тогда, когда он защищает правило, делает сравнение
    явным или заметно улучшает язык модели.
11. Bounded context не использует напрямую агрегаты, репозитории и доменные типы
    другого контекста.

Не требуется:

* создавать интерфейс для каждого класса;
* оборачивать каждый `string` в Value Object;
* использовать domain event для каждого изменения поля;
* строить отдельный слой абстракций без реальной сменяемости или бизнес-смысла;
* добиваться полной независимости Domain от ORM ценой существенного усложнения.

Doctrine attributes в доменных сущностях допустимы как прагматичный компромисс,
пока они не начинают определять бизнес-поведение.

# Взаимодействие bounded contexts

Контексты должны взаимодействовать только через явно определённые границы.

Предпочтительный способ взаимодействия — domain events и integration messages,
если use case допускает асинхронность и eventual consistency.

Если для выполнения use case необходим немедленный ответ другого контекста,
допускается синхронное взаимодействие через application-port:

* порт принадлежит контексту-потребителю и располагается в его Application layer;
* порт описывает потребность потребителя, а не API чужого репозитория;
* контракт возвращает только необходимые потребителю DTO, snapshot, скаляры или
  его собственные типы;
* реализация порта располагается в Infrastructure/Integration и адаптирует модель
  контекста-поставщика;
* Application и Domain потребителя не импортируют агрегаты, репозитории,
  исключения и доменные типы поставщика;
* контекст-поставщик не должен зависеть от порта потребителя.

Прямой доступ к таблицам другого контекста из application или domain кода не
допускается. Общая база данных в модульном монолите является инфраструктурной
деталью и не отменяет границы контекстов.

Порт не следует вводить автоматически для любой связи. Сначала рассматриваются
передача идентификатора, событие, integration message или локальная read-модель.
Синхронный порт используется, когда без немедленного ответа нельзя надёжно
продолжить текущий use case.

# Карта bounded contexts

## aeroflow-core

### User Access

Статус: реализован.

Отвечает за:

* регистрацию пользователя;
* аутентификацию;
* выдачу и ротацию токенов;
* завершение пользовательской сессии;
* роли и доступ.

### Flight Operations

Статус: частично реализован (`FlightDefinition` и `FlightOccurrence`).

Отвечает за:

* каталог рейсов;
* будущее расписание;
* конкретные operational рейсы (`FlightOccurrence`);
* регистрацию;
* посадку;
* прибытие;
* назначение выхода и стоек регистрации;
* оркестрацию запуска рейсового объявления как действия над `FlightOccurrence`.

### Airport Directory

Статус: реализован внутри Flight Operations.

`Airport` является отдельным справочным aggregate root. Он хранит IATA-код,
название аэропорта, город, страну и активность. `FlightDefinition` ссылается на
аэропорты через `AirportCode`, а не через ORM-граф объектов.

### Announcements

Статус: частично реализован (доменная модель ручного рейсового объявления).

Отвечает за:

* причину и содержание объявления;
* тип объявления;
* связь объявления с рейсом;
* шаблоны;
* языковые варианты;
* подготовку аудиопоследовательности;
* запрос воспроизведения.

### Audio Catalog

Статус: частично реализован (`AudioAsset` metadata catalog).

Отвечает за:

* метаданные аудиофайлов;
* языки;
* категории;
* совместимость с библиотекой старой САО;
* доступность файлов для формирования объявлений.

### Localization

Статус: реализован как read-only справочник из конфигурации.

Отвечает за:

* список поддерживаемых языков AeroFlow;
* человекочитаемые названия языков для web-интерфейса;
* единый источник вариантов для выбора `LanguageCode` в формах.

На текущем этапе языки не являются aggregate root и не хранятся в базе данных.
Write API других контекстов продолжает принимать `languageCode` как строковый
BCP 47-совместимый код, который валидируется через `LanguageCode`.

### Audit

Статус: планируется.

Отвечает за неизменяемую историю значимых действий пользователей и системы.
Audit является supporting subdomain и может строиться из доменных событий.

## aeroflow-playback

### Playback

Статус: планируется.

Отвечает только за `PlaybackJob`, очередь, приоритеты, повторы и emergency mode.

Контекст не знает о `FlightDefinition`, `FlightSchedule`, `FlightOccurrence`,
`Gate` и `CheckInCounter`.

## aeroflow-agent

### Audio Execution

Статус: реализованы первый (задача 016) и второй (задача 019) срезы.

Отвечает за физическое выполнение команд воспроизведения и отправку результата.
Не содержит бизнес-правил аэропорта.

Решения первого среза:

* стек — C#/.NET Worker Service: агент — on-prem процесс на машине у
  аудиовыхода (Windows, как у старой САО); self-contained бинарник;
* транспорт playback ↔ agent — RabbitMQ (закрыт открытый вопрос
  architecture.md): очередь `agent_commands` (команда `audio.play_sequence` с
  дискриминатором `command` в теле — ADR 002 с первого дня) и `agent_events`
  (`audio.playback_started|completed|failed` с дискриминатором `event`);
* файлы — download-контракт Audio Catalog:
  `GET /internal/v1/audio-assets/{id}/file` + локальный дисковый кэш агента по
  `audioAssetId`; авторизации между сервисами нет — сетевое доверие локальной
  установки, известный временный компромисс;
* воспроизведение — системный плеер (`ffplay`) дочерним процессом; режим
  `Simulate` для сред без аудиоустройства;
* агент без БД: дедупликация повторной команды best-effort (память + маркеры в
  кэше); строгая идемпотентность остаётся в playback (`CompletePlaybackJob`);
* guard свежести: команда старше `MaxCommandAgeSeconds` (по `occurredAt`
  контракта, по умолчанию 300 с) не воспроизводится, а фейлится с reason
  `stale command` — durable-очередь переживает офлайн агента днями, а
  объявление аэропорта имеет смысл минуты; отказ виден на экране очереди;
* события агента `completed`/`failed` в playback ложатся в идемпотентные
  `CompletePlaybackJob`/`FailPlaybackJob` (шов задачи 015); `started`
  логируется; recovery зависшего `Playing` после смерти агента — будущий срез.

Решения второго среза (задача 019 — остановка звучащего):

* очередь `agent_commands` получает второй тип команды — `audio.stop` с
  `jobId`, различаются дискриминатором `command` в теле;
* агент подтверждает `audio.play_sequence` сразу после запуска фонового
  воспроизведения (не после завершения) и хранит один активный `jobId` с
  `CancellationTokenSource`, чтобы `audio.stop` мог быть обработан, пока играет
  долгая последовательность; второй `play` не запускается, пока активный job не
  сообщил терминальный результат;
* отмена процесса плеера не ошибка: дочерний процесс убивается целиком
  (process tree), агент публикует `audio.playback_interrupted`, а не
  `audio.playback_failed`; simulate-режим корректно реагирует на cancellation
  token;
* `Pause/Resume/SetVolume` — будущие команды той же очереди (emergency mode).

# Реализованная модель: User Access

## User

`User` является aggregate root.

Состояние:

* `id` — UUID, создаётся при регистрации;
* `email` — идентификатор пользователя;
* `password` — только хеш пароля;
* `roles` — назначенные роли, при чтении всегда включает `ROLE_USER`.

Поведение:

* `register(email, passwordHash)` — создаёт пользователя;
* `pullEvents()` — передаёт накопленные domain events application layer.

Инварианты:

* email уникален в системе;
* открытый пароль не хранится;
* каждый пользователь имеет `ROLE_USER`;
* событие регистрации содержит уже созданный `userId`.

Проверка уникальности email требует репозитория, поэтому выполняется application
handler до создания и сохранения пользователя. Ограничение уникальности в базе
остаётся последней защитой от конкурентной регистрации.

События:

* `UserRegistered`;
* `UserLoggedIn`;
* `UserLoggedOut`.

`UserRegistered` создаётся агрегатом. События входа и выхода создаются application
layer, потому что относятся к сценарию аутентификации, а не к изменению состояния
самого `User`.

## RefreshToken

`RefreshToken` хранится отдельно и имеет самостоятельный жизненный цикл.
Практически он является отдельной транзакционной границей аутентификации, даже
если не наследуется от общего `AggregateRoot`.

Состояние:

* `id`;
* ссылка на `User`;
* `tokenHash`;
* `expiresAt`;
* `revokedAt`;
* `replacedByHash`;
* `createdAt`.

Инварианты:

* открытое значение refresh token не хранится;
* hash токена уникален;
* токен активен, только если не отозван и не истёк;
* отзыв токена идемпотентен;
* ротация отзывает старый токен и создаёт новый;
* отозванный или истёкший токен нельзя использовать повторно.

## Use cases

### Register User

1. Проверить уникальность email.
2. Захешировать пароль.
3. Создать и сохранить `User`.
4. Опубликовать `UserRegistered`.
5. Выдать access token и refresh token.

### Login User

1. Найти пользователя по email.
2. Проверить пароль.
3. Выдать пару токенов.
4. Опубликовать `UserLoggedIn`.

### Refresh Token

1. Найти refresh token по hash.
2. Проверить, что он активен.
3. Отозвать текущий токен.
4. Создать и сохранить новый токен.
5. Выдать новую пару токенов.

### Logout User

1. Найти refresh token по hash.
2. Проверить, что он активен.
3. Отозвать токен.
4. Опубликовать `UserLoggedOut`.

# Целевая модель: Flight Operations

## FlightDefinition

Статус: реализован.

`FlightDefinition` — постоянная карточка в справочнике рейсов. Администратор
управляет справочником, а диспетчер выбирает активную карточку в отдельном рабочем
сценарии, когда нужно запустить связанное с рейсом объявление.

Карточка не представляет конкретный вылет и не создаётся заново каждый день.

Состояние:

* `id`;
* номер рейса;
* направление: вылет или прибытие;
* аэропорт отправления;
* аэропорт назначения;
* признак активности;
* даты создания и изменения.

Инварианты:

* номер и коды аэропортов имеют допустимый формат;
* аэропорты отправления и назначения различаются;
* бизнес-ключ карточки уникален;
* неиспользуемая карточка деактивируется, а не удаляется физически;
* повторная активация или деактивация идемпотентна.

Domain events:

* `FlightDefinitionCreated`;
* `FlightDefinitionUpdated`;
* `FlightDefinitionActivated`;
* `FlightDefinitionDeactivated`.

Текущий сценарий:

```text
FlightDefinition
    -> FlightOccurrence
    -> Announcement
```

## FlightSchedule

Статус: возможное будущее расширение.

`FlightSchedule` задаёт правило времени и повторяемости для `FlightDefinition`.
Оно добавляется только при появлении автоматического или полуавтоматического
режима и не требуется для текущей ручной работы диспетчера.

## FlightOccurrence

Статус: реализован как core-срез operational рейса.

`FlightOccurrence` представляет конкретный operational рейс в определённую дату.
Он нужен, чтобы независимо хранить:

* состояние регистрации и посадки;
* назначенные стойки и выход;
* связь с последним объявлением;
* историю действий и объявлений конкретного вылета или прилёта.

Lifecycle вылета:

```text
scheduled
  -> check_in_open
  -> check_in_closed
  -> boarding
  -> completed
```

Lifecycle прилёта:

```text
scheduled
  -> arrival_announced
  -> completed
```

Отмена поддерживается как переход в `cancelled`. Задержка и фактическое время
остаются будущими расширениями.

Состояние:

* `id`;
* `flightDefinitionId`;
* `source`: `manual` или `schedule`;
* `direction`: совпадает с направлением `FlightDefinition`;
* `operationalDate`;
* `sequenceNumber`;
* snapshots номера рейса и кодов аэропортов;
* `status`;
* snapshots выбранных стоек и выхода;
* `lastAnnouncementId`;
* даты создания, изменения, завершения и отмены.

Инварианты:

* occurrence создаётся только для активного `FlightDefinition`;
* `direction` occurrence совпадает с направлением `FlightDefinition`;
* бизнес-ключ `flightDefinitionId + operationalDate + source + sequenceNumber`
  уникален;
* нельзя открыть регистрацию дважды, продолжить или закрыть её до открытия,
  начать посадку до закрытия регистрации;
* нельзя объявить прибытие для вылетающего occurrence и открыть регистрацию или
  посадку для прибывающего;
* завершённый или отменённый occurrence нельзя менять;
* следующий manual-occurrence по той же карточке за ту же `operationalDate`
  создаётся только когда последний прогон карточки достиг финального статуса
  своего цикла — `boarding` для вылета или `arrival_announced` для прилёта (а
  также терминальные `completed`/`cancelled`); пока последний прогон не дошёл до
  финального статуса, новый прогон отклоняется;
* выбранные стойки и выход сохраняются как snapshot на момент перехода;
* `lastAnnouncementId` — голый идентификатор объявления (правило #6), а не
  ORM-связь с агрегатом `Announcement`.

Domain events:

* `FlightOccurrenceCreated`;
* `CheckInOpened`;
* `CheckInContinued` (`@deprecated`, задача 020 — см. ниже);
* `CheckInClosed`;
* `BoardingStarted`;
* `ArrivalAnnounced`;
* `FlightOccurrenceCompleted`;
* `FlightOccurrenceCancelled`.

Эти события — факты уже состоявшегося перехода. Они потребляются audit и
read-моделью. Они **не** используются как триггер создания основного объявления:
создание объявления является частью самого действия диспетчера, а не реакцией на
событие перехода (см. ниже).

`continueCheckIn()`/`CheckInContinued` были задуманы как trigger будущего
per-tick process manager «продолжение регистрации» — каждый тик создаёт новый
`Announcement`. Задача 020 сознательно сменила модель: повтор владеет playback и
крутит **один** `Announcement` по кругу (см. «Пятый срез» `PlaybackJob` и
раздел `Announcements` ниже), поэтому per-tick process manager не строится, а
`continueCheckIn()`/`CheckInContinued` помечены `@deprecated` и не вызываются ни
одним потоком — сохранены только для обратной совместимости метода на агрегате,
без нового поля состояния.

### Создание occurrence

Manual occurrence создаётся тремя use case. `CreateManualFlightOccurrence`
строгий: на повтор бизнес-ключа отвечает конфликтом. `EnsureManualFlightOccurrence`
идемпотентный: возвращает существующий occurrence по бизнес-ключу (`source=manual`,
`sequenceNumber=1`) или создаёт новый. Идемпотентный вариант обслуживает
definition-driven панель диспетчера, где occurrence обеспечивается перед **первым**
действием над карточкой; повторный вызов и двойной клик не создают дубликат.

`StartNextManualFlightOccurrence` обслуживает **повторный прогон** той же карточки
в тот же день: `sequenceNumber` назначает сервер (`max + 1` по
`flightDefinitionId + operationalDate + source=manual`), а не клиент. Use case —
**не** идемпотентный по фиксированному ключу; защиту от дубля и двойного клика даёт
доменное состояние: новый прогон создаётся только когда последний прогон карточки
достиг финального статуса своего цикла (`boarding`/`arrival_announced`, либо
терминальных `completed`/`cancelled`), иначе команда отклоняется конфликтом.
Последний прогон грузится на запись, поэтому параллельные команды не выделяют один
`sequenceNumber`, а уникальный индекс бизнес-ключа остаётся последней защитой. Так
ручной поток допускает несколько occurrence в день без отдельного idempotency key.

`completed`/`cancelled` пока недостижимы в ручном потоке (явного действия
завершения/отмены нет), поэтому фактическим триггером нового прогона служит
финальный статус действия цикла — `boarding` или `arrival_announced`. Доковый
переход `... -> completed` остаётся будущим расширением; когда появится действие
завершения, оно лишь расширит уже существующий набор финальных статусов.

### Запуск рейсового объявления

Запуск рейсового объявления — это **действие над `FlightOccurrence`**, а не над
`FlightDefinition`. `FlightDefinition` остаётся только шаблоном/карточкой.
Одно действие диспетчера атомарно: продвигает lifecycle occurrence и создаёт
`Announcement`.

Правила:

* оркестрация принадлежит `Flight Operations\Application` (per-action use cases
  открытия/закрытия регистрации, посадки, прибытия), потому что владельцем
  состояния перехода является `FlightOccurrence`;
* агрегат `FlightOccurrence` — единственный авторитет допустимости перехода;
  дублировать матрицу direction + status в другом контексте нельзя;
* готовность объявления (резолв шаблона в `AudioSequence`) является
  **предусловием** перехода: если объявление нельзя подготовить, переход
  отклоняется и ничего не персистится;
* переход occurrence и создание `Announcement` фиксируются в **одной локальной
  транзакции** Core; нельзя сохранить `Announcement` без применённого перехода и
  наоборот;
* `Flight Operations` создаёт `Announcement` через принадлежащий ему порт в
  `FlightOperations\Application\Port\Announcements`; порт возвращает только
  `announcementId` и нужные скаляры/snapshot;
* `Announcements` не обращается к write-side `FlightOccurrence` и не мутирует
  чужой агрегат.

Тот же оркестратор (`LaunchOccurrenceAnnouncementHandler`) в той же локальной
транзакции ведёт и серию повтора «продолжение регистрации» (задача 020):
открытие регистрации, помимо одноразового `check_in_opening`, запускает
continuation-серию через порт, если для рейса включён и настроен continuation-
конфиг (иначе — no-op, рейс без настроенного повтора открывается как раньше);
закрытие регистрации завершает активную серию тем же способом. Ради этого порт
`FlightOperations\Application\Port\Announcements\AnnouncementLauncherInterface`
переименован в `OccurrenceAnnouncementPort`: с двумя операциями одного рода
(синхронные, co-committing мутации `Announcements` внутри occurrence-транзакции)
имя «Launcher» стало слишком узким. Интерфейс остаётся единым — у него один
потребитель (сам оркестратор), поэтому ISP-разведение на два порта не даёт
изоляции, только церемонию.

```text
dispatcher action on FlightOccurrence
    -> FlightOccurrence transition (authority)  ─┐ одна локальная транзакция
    -> create Announcement (через порт)         ─┘
    -> domain events (факты, после коммита)
    -> RequestAnnouncementPlayback (async, outbox, отдельная граница)
```

Текущая связь:

```text
FlightDefinition (шаблон)
    -> FlightOccurrence (действие диспетчера)
    -> Announcement
```

Будущая связь с расписанием:

```text
FlightDefinition
    -> FlightSchedule
    -> FlightOccurrence
```

## Gate и CheckInCounter

Статус: реализованы как отдельные справочные aggregate roots.

Обе сущности содержат UUID, уникальный строковый код, отображаемое имя, порядок,
активность и даты изменения. Поддерживаются числовые и составные коды, например
`1`, `12`, `A5`, `B-2`.

Диспетчер передаёт идентификаторы, а `Announcement` сохраняет ID и код как
неизменяемый snapshot. Деактивированную запись нельзя выбрать для нового
объявления.

Справочники не моделируют пока:

* конфликт назначений;
* доступность по времени;
* терминалы и зоны;
* блокировка или техническое обслуживание.

# Частично реализованная модель: Announcements

## Announcement

Статус: реализован с persistence, application layer и подготовкой
`AudioSequence`.

`Announcement` отвечает на вопросы:

```text
Что должно быть объявлено?
Почему это объявление создано?
На каких языках оно должно прозвучать?
```

Реализованное состояние:

* `id`;
* тип;
* связанный `flightDefinitionId`;
* связанный `flightOccurrenceId`, если объявление создано для operational рейса;
* snapshots выбранных стоек или выхода;
* непустой упорядоченный набор языков;
* неизменяемая подготовленная `AudioSequence`;
* состояние `Prepared` или `Cancelled`;
* дата создания и дата отмены;
* интервал повтора `repeatEveryMinutes` и дата окончания серии повтора
  `repeatEndedAt` — только для повторяемого типа (продолжение регистрации, задача
  020), у остальных типов всегда `null`.

Реализованные типы:

* начало регистрации;
* продолжение регистрации (единственный повторяемый тип, задача 020);
* окончание регистрации;
* приглашение на посадку;
* прибытие.

Реализованные инварианты:

* рейсовой тип содержит идентификатор `FlightDefinition`, но не ORM-связь или
  объект другого агрегата;
* принадлежность объявления к рейсу или всему аэропорту определяется его типом,
  без отдельного `AnnouncementSubject`;
* ссылка на рейс неизменяема после создания;
* объявление о **начале** регистрации содержит необходимые данные о стойках;
  объявление об окончании регистрации стойки не требует;
* объявление о посадке содержит выход;
* объявление содержит хотя бы один язык без дубликатов;
* порядок языков сохраняется для будущей подготовки и воспроизведения;
* после подготовки языки нельзя изменить отдельно от последовательности;
* новое объявление создаётся только после полного разрешения шаблона;
* отсутствие стойки, выхода, asset или prompt блокирует создание;
* отмена идемпотентна и публикует событие только при первом переходе;
* отменённое объявление нельзя вернуть в активное состояние или изменить его
  языки;
* только повторяемый тип принимает `repeatEveryMinutes`; для прочих типов
  переданный интервал игнорируется (не является ошибкой);
* `endRepeat()` идемпотентен и публикует `AnnouncementRepeatEnded` только при
  первом переходе; не меняет `status` — объявление остаётся `Prepared`.

Реализованные domain events:

* `AnnouncementCreated`;
* `AnnouncementCancelled`;
* `AnnouncementRepeatEnded` (задача 020).

Проверка существования и активности `FlightDefinition`, persistence, HTTP API и
управление через HTTP API реализованы. Frontend относится к следующей задаче.

Рейсовое объявление создаётся как часть действия диспетчера над
`FlightOccurrence`. Оркестрацию ведёт `Flight Operations\Application`: он
продвигает occurrence и в той же локальной транзакции создаёт `Announcement`
через принадлежащий `Flight Operations` порт в
`FlightOperations\Application\Port\Announcements`. Создание объявления является
предусловием перехода: если шаблон не резолвится в `AudioSequence`, переход
отклоняется и `Announcement` не создаётся.

`Announcements` остаётся поставщиком: он собирает и сохраняет `Announcement` по
переданному occurrence-snapshot и возвращает `announcementId` и нужные скаляры.
`Announcements` не обращается к write-side `FlightOccurrence` и не мутирует чужой
агрегат. Реализация порта `Flight Operations` располагается в
`Announcements\Infrastructure\Integration` — поставщик не зависит от порта
потребителя.

Для справочных сведений `Announcements` использует собственные read-only порты к
`Flight Operations` и `Audio Catalog` в `Announcements/Application/Port`.
Реализация порта может читать чужую модель, но application и domain слои
`Announcements` не используют чужой агрегат, репозиторий или доменные типы
напрямую.

`FlightAnnouncementConfig` содержит языковые `AnnouncementVariant`. Каждый
вариант состоит из упорядоченных сегментов:

* `audio_asset`;
* `dynamic_slot`;
* `pause`;
* `text`.

Реализованные dynamic slots: `check_in_counters` и `gate_code`. Текст хранится
как утверждённое содержимое, но без TTS делает конфигурацию неготовой.

`Announcements` предоставляет read-only запрос настроенных языков объявления для
пары рейс × тип (`ListConfiguredAnnouncementLanguages`): он возвращает коды
enabled-вариантов enabled-конфига в порядке `sortOrder` либо пустой список. Панель
диспетчера использует его, чтобы по умолчанию выбрать все настроенные языки
объявления. Это локальный CQRS-read внутри контекста-владельца конфигурации; он не
загружает агрегат `FlightOccurrence` и не пересекает границу сервиса.

При запуске application layer разрешает сегменты через consumer-owned ports к
Flight Operations и Audio Catalog. Snapshots и итоговая последовательность
хранятся в PostgreSQL JSON, поэтому последующее изменение справочника, prompt или
шаблона не меняет уже созданное объявление.

`Announcement` не должен считать legacy-файл обязательным источником. Независимо
от источника, в playback передаётся уже подготовленная аудиопоследовательность, а
не текст или внутренние сущности Core.

Для языка используется Value Object `LanguageCode` с BCP 47-совместимым
обозначением. Поддерживаемые языки не зашиваются в enum и могут управляться
конфигурацией или справочником.

`Announcement` создаётся в Core синхронно и в одной транзакции с переходом
occurrence. Отправка в playback — отдельная асинхронная граница ниже по течению:
после коммита локальной транзакции application layer контекста `Announcements`
публикует integration command `RequestAnnouncementPlayback` в `aeroflow-playback`,
а `PlaybackJob` создаётся уже в playback eventual-образом. Нельзя путать эти два
«создания»: `Announcement` принадлежит Core, `PlaybackJob` — playback.

Издателем `RequestAnnouncementPlayback` является контекст-владелец `Announcement`
и готовой `AudioSequence` — `Announcements`, а не `Flight Operations`. Публикация
выполняется как реакция application layer на доменный факт `AnnouncementCreated`
**после** коммита (через тот же post-commit механизм, что и остальные доменные
события: события буферизуются в transaction scope и отдаются только после успешного
коммита). Так `Flight Operations` не знает про playback, а контракт принадлежит
владельцу аудиопоследовательности.

Транспорт между Core и `aeroflow-playback` — RabbitMQ (Symfony Messenger,
AMQP-транспорт). Durable transactional outbox в первой версии **отложен**: пока
используется post-commit публикация из application layer без отдельной outbox-таблицы.
Это осознанный временный компромисс с известным риском потери сообщения при крахе
процесса между коммитом и публикацией; durable outbox вводится отдельным шагом и не
меняет доменную модель (точка публикации остаётся в application layer).

Контракт `RequestAnnouncementPlayback` нейтрален и не содержит аэропортовых
сущностей: `command` (дискриминатор типа в теле — ADR 002, введён задачей 018),
`messageId` (идемпотентность), `correlationId` (= `announcementId`),
`announcementId`, `type`, `priority`, готовая `audioSequence`, `repeatRule`
(пока `null`), `occurredAt`, `schemaVersion` (текущая версия — 2, добавление
`command`). `audioSequence` берётся из `Announcement` как есть; playback не
получает шаблоны, prompts, стойки или выходы.

Для отмены используется `CancelAnnouncementPlayback` (реализовано задачей 018).
Издатель тот же — `Announcements`, post-commit реакция на доменный факт
`AnnouncementCancelled`: агрегат публикует событие только при первом переходе в
`Cancelled`, поэтому повторная отмена не дублирует команду. Контракт:
`command: announcement_playback.cancel`, `messageId`, `correlationId`
(= `announcementId`), `announcementId`, `occurredAt`, `schemaVersion` — без
аудиопоследовательности. Отмена объявления не трогает `FlightOccurrence`:
состоянием рейса владеет Flight Operations, срез отменяет только объявление и
его ожидающее задание.

Тип продолжения регистрации (`check_in_continuation`, задача 020) — единственный
повторяемый (`AnnouncementType::isRepeatable()`): `Announcement` хранит
`repeatEveryMinutes` (из `FlightAnnouncementConfig`, только для этого типа) и
кладёт его в `RequestAnnouncementPlayback.repeatRule` как `{everyMinutes: N}`.
Повтором на всех тиках управляет playback вокруг **одного и того же**
`Announcement` — per-tick создание объявления и событие `CheckInContinued`
(целевая модель до 020) не реализуются; `FlightOccurrence::continueCheckIn()` и
`CheckInContinued` помечены `@deprecated` и не вызываются ни одним потоком (см.
раздел `FlightOccurrence`). Серию завершает `endRepeat()` — идемпотентный переход
(как `cancel()`), публикующий `AnnouncementRepeatEnded` только при первом
переходе; сам `Announcement` остаётся `Prepared`, это не отмена. `Announcements`
post-commit реагирует на `AnnouncementRepeatEnded` и публикует
`StopAnnouncementRepeat` (`command: announcement_playback.stop_repeat`, тот же
`messageId`/`correlationId`/`announcementId`/`occurredAt`/`schemaVersion`
контракт, что у cancel и stop — четвёртый тип на очереди `announcement_playback`,
ADR 002). Оркеструет старт и конец серии `Flight Operations` через тот же порт,
что и запуск обычного объявления (см. `OccurrenceAnnouncementPort` ниже) — открытие
регистрации атомарно создаёт оба объявления (`check_in_opening` разово,
`check_in_continuation` с `repeatRule`, если для рейса включён и настроен
continuation-конфиг), закрытие атомарно завершает серию.

# Частично реализованная модель: Audio Catalog

## AudioAsset

Статус: реализован минимальный aggregate root для конфигураций объявлений.

`AudioAsset` представляет доступный системе аудиоматериал, но не воспроизводит его.

Предполагаемое состояние:

* `id`;
* имя или бизнес-код;
* категория;
* язык;
* расположение файла;
* формат и длительность, если известны;
* источник: legacy, uploaded или generated;
* состояние доступности.

На текущем этапе реализованы стабильный идентификатор, имя, язык, признак
активности, MIME type, размер и непрозрачный ключ файлового хранилища. Admin API
позволяет загрузить WAV, MP3 или OGG до 50 МБ и получить список активных assets
для выбора в `FlightAnnouncementConfig`. Файлы хранятся вне публичного
web-каталога под сгенерированными именами.

Категории, длительность, отдельный download contract и импорт legacy-каталогов
остаются будущими расширениями.

Domain event:

* `AudioAssetUploaded`.

## Генерация assets через TTS

Статус: решение принято, реализация — отдельным таском.

`AudioAsset` со `source = generated` создаётся синтезом речи из текста. Это
закрывает уже существующий разрыв: сегмент `text` в `AnnouncementVariant`
хранится как утверждённое содержимое, но без озвучивания делает конфигурацию
неготовой (см. `FlightAnnouncementConfig`). TTS превращает `text`-сегмент в
`generated` `AudioAsset`, после чего конфиг становится готовым и резолвится в
`AudioSequence` наравне с загруженными и legacy-файлами.

Принятые решения:

* синтез — **отдельный stateless-сервис** (`aeroflow-tts`), нейтральный к домену:
  знает только `text`, `language`, `voice`; аэропортовые понятия (объявление,
  рейс, тип, приоритет, стойка, выход) внутрь не протекают, поэтому сервис
  переиспользуем и не является скрытым куском Audio Catalog. Контракт —
  `POST /v1/synthesize { text, language, voice? } -> audio/wav`, плюс
  `GET /v1/voices` и `GET /health`. У сервиса нет БД и доменных инвариантов;
* движок синтеза — деталь реализации сервиса за его HTTP-контрактом (кандидат
  первого шага — Piper: нативный оффлайн-бинарник, ru/en голоса; замена на другой
  движок не меняет контракт). Стек сервиса выбирается на этапе реализации;
  self-hosted/оффлайн — принятое ограничение (on-prem, сетевое доверие локальной
  установки, как у остальных компонентов);
* Core обращается к сервису **синхронно по HTTP** через consumer-owned порт
  `TextToSpeechPort` в `AudioCatalog\Application\Port`; реализация
  (`HttpTtsAdapter`) — в `AudioCatalog\Infrastructure\Integration`. Синхронный
  порт оправдан: генерация — предусловие готовности конфигурации, нужен
  немедленный ответ (не async-граница, не RabbitMQ);
* момент генерации — **пре-шаг при сохранении текста** конфигурации, не горячий
  путь запуска объявления. Недоступность TTS блокирует только сохранение шаблона,
  а не атомарный переход `FlightOccurrence`; задержка синтеза не попадает в
  критический путь диспетчерского действия;
* **кэш и идемпотентность генерации — в Core** (Audio Catalog), а не в сервисе:
  по ключу `hash(text + language + voice + версия модели)` Core решает,
  переиспользовать существующий `AudioAsset` или звать сервис. Так сервис
  остаётся stateless. Изменение текста даёт новый asset; прежний деактивируется, а
  не удаляется физически (инвариант «удаление используемого asset не должно молча
  ломать существующие объявления»);
* выбор голоса под тип объявления делает Core и передаёт `voice` в контракт;
  сервис не догадывается о назначении текста.

Архитектурно `aeroflow-tts` — инфраструктурный сервис в роли Audio Catalog, а
**не новый bounded context**: у него нет своего домена, агрегатов и БД. Он
дополняет Audio Catalog возможностью генерации, как download-контракт дополняет
его раздачей файлов.

## AudioPrompt

Статус: реализован.

`AudioPrompt` связывает семантический ключ с активным `AudioAsset`:

```text
kind + value + languageCode -> audioAssetId
```

Поддерживаются `check_in_counter_code` и `gate_code`. Имя файла и storage key не
участвуют в поиске. Для составного кода требуется точный prompt; автоматическая
композиция букв и чисел пока не выполняется.

Инварианты:

* asset имеет стабильный идентификатор независимо от физического пути;
* недоступный asset нельзя включать в новое объявление;
* удаление используемого asset не должно молча ломать существующие объявления;
* legacy-файлы старой САО должны быть представлены через тот же доменный контракт.

Категории и языки могут оставаться справочниками, пока у них нет собственного
поведения.

`AudioPrompt` координируется через command/query handlers на application bus,
а не через CRUD-сервис: `CreateAudioPromptCommand`, `UpdateAudioPromptCommand`,
`ChangeAudioPromptStatusCommand` и `ListAudioPromptsQuery` возвращают
`AudioPromptResult`. Это оправдано тем, что у `AudioPrompt` есть жизненный цикл и
инварианты (валидация и нормализация `value`, уникальность активной комбинации
`kind + value + languageCode`, запрет ссылки на недоступный или неактивный
`AudioAsset` — межагрегатная проверка), а также доменные события
`AudioPromptCreated`, `AudioPromptUpdated`, `AudioPromptActivated`,
`AudioPromptDeactivated`.

Известное расхождение: `Gate` и `CheckInCounter` имеют ту же форму (aggregate root
с инвариантами и событиями, координируемый тонкой application-директорией
`GateDirectory` / `CheckInCounterDirectory`), но пока сознательно оставлены
директориями. Это осознанный переходный компромисс, а не принципиальная граница;
приведение их к тому же стилю, что и `AudioPrompt`, вынесено в открытые решения.

# Целевая модель: Playback

## PlaybackJob

Статус: реализованы первый, второй, третий, четвёртый и пятый тонкие срезы в
`aeroflow-playback`. Первый — приём `RequestAnnouncementPlayback` и
идемпотентное создание задания. Второй (задача 015) — последовательная
приоритетная очередь, жизненный цикл задания и обратные integration events в
core. Третий (задача 018) — отмена ожидающего задания по
`CancelAnnouncementPlayback`. Четвёртый (задача 019) — остановка звучащего
задания: `StopAnnouncementPlayback` от core, `audio.stop` агенту, переход
`Playing -> Interrupted` только по факту `audio.playback_interrupted`. Пятый
(задача 020) — повтор continuation-задания вокруг самого себя: рождение в
`Scheduled`, re-arm `Playing -> Scheduled` на каждом тике, delayed self-wake
без cron и `StopAnnouncementRepeat` от core на закрытии регистрации.
Приоритетное прерывание и emergency mode — следующие срезы, пока планируются.

`PlaybackJob` не является частью Flight Operations и не содержит аэропортовые
сущности.

### Первый срез

Первый срез реализует минимальную приёмную сторону:

* `aeroflow-playback` поднимается как отдельный Symfony-сервис (Messenger,
  Doctrine, PostgreSQL);
* consumer `RequestAnnouncementPlayback` создаёт и сохраняет `PlaybackJob` в
  статусе `Pending`;
* создание **идемпотентно по `messageId`**: повторная доставка не создаёт
  дубликат, а возвращает уже достигнутый результат;
* `priority` берётся из контракта; для рейсового объявления — `100`;
* `audioSequence` и `repeatRule` сохраняются как есть.

Вне первого среза: приоритетная очередь и прерывание, фоновая музыка, повторы,
emergency mode, `CancelAnnouncementPlayback`, обратные integration events
playback → core и `aeroflow-agent`.

### Второй срез: очередь и жизненный цикл (задача 015, реализован)

Решения второго среза:

* очередь строго последовательная, **без прерывания**: обычные объявления не
  прерывают друг друга (прерывает только будущий emergency mode); активное
  задание доигрывается, следующее выбирается после его завершения;
* порядок выбора: `priority` по убыванию (100 > 50 > 0), при равенстве — FIFO по
  `createdAt`;
* single-active инвариант: не более одного задания в статусе `Playing` на один
  audio output; в первой версии audio output один (глобальный);
* переходы: `Pending -> Playing -> Completed | Failed`; `Scheduled`,
  `Interrupted`, `Cancelled` вводятся следующими срезами;
* `Completed`/`Failed` терминальны; повторное завершение идемпотентно;
* завершение моделируется application-командой `CompletePlaybackJob` /
  `FailPlaybackJob` — она представляет факт «audio output сообщил результат».
  Будущее событие агента `PlaybackCompleted`/`PlaybackFailed` станет источником
  этой команды без изменения доменной модели. До появления агента команду в
  тестах подаёт стаб; в проде очередь останавливается на первом `Playing` —
  осознанное ограничение тонкого среза;
* outbound-команда агенту (`PlayAudioSequence`) отложена до agent-среза; переход
  в `Playing` не порождает внешней команды;
* playback публикует обратные integration events в core:
  `AnnouncementPlaybackQueued` (создание задания), `AnnouncementPlaybackStarted`
  (`Pending -> Playing`), `AnnouncementPlaybackCompleted`
  (`Playing -> Completed`) и `AnnouncementPlaybackFailed`
  (`Playing -> Failed`, с `reason`; добавлено задачей 017 — без него упавшее
  задание навсегда выглядело бы играющим на экране очереди). Контракт нейтрален:
  `event` (дискриминатор факта в теле сообщения — получатель не зависит от
  PHP-классов издателя), `messageId`, `correlationId` = `announcementId`,
  `announcementId`, `jobId`, `occurredAt`, `schemaVersion`;
* core-сторона минимальна: `Announcements` идемпотентно (по `messageId`)
  принимает события и фиксирует факт приёма; статусы `Announcement` не меняются
  (расширение его жизненного цикла — отдельная задача). Поверх receipts построен
  локальный CQRS-read `ListPlaybackQueue` (задача 017): экран очереди диспетчера
  (наследник окна «Статус» старой САО) — играющее, ожидающие, недавние; playback
  остаётся источником истины порядка, read-модель eventual;
* фоновая музыка: приоритет `0` остаётся правилом порядка; продюсера музыки и
  idle-состояния нет.

### Третий срез: отмена ожидающего (задача 018, реализован)

Решения третьего среза:

* на очереди `announcement_playback` появился второй тип сообщения — исполнен
  уровень 1 и 3 ADR 002: дискриминатор `command` в теле
  (`announcement_playback.request` | `announcement_playback.cancel`), `match` в
  сериализаторе с ошибкой на неизвестный тип; тело без `command` переходно
  трактуется как request (сообщение старого контракта из durable-очереди) и
  этот fallback удаляется следующим контрактным изменением;
* отменяется **только `Pending`**: переход `Pending -> Cancelled` идемпотентен
  (повторная отмена — no-op без события). Job в `Playing` или терминальном
  статусе — «опоздавшая отмена»: команда подтверждается без изменения состояния
  и без события; остановка звучащего — отдельный будущий срез. Расхождение
  «объявление в core отменено, а звук играет» осознанное и видно на экране
  очереди;
* поиск задания — по `announcementId` с write lock (та же транзакционная манера,
  что выбор следующего задания), чтобы отмена не гонялась с запуском этого же
  job; неизвестный `announcementId` подтверждается с логом. Гонка «cancel
  обогнал request» практически исключена: оба типа идут одной очередью и порядок
  сохраняется;
* при первом переходе публикуется обратное событие
  `AnnouncementPlaybackCancelled` (`event: announcement_playback.cancelled`,
  контракт как у остальных); приёмная сторона core generic по имени события и
  не менялась, read-модель `ListPlaybackQueue` показывает строку в «Недавних»
  как `cancelled`;
* отдельного продвижения очереди после отмены не требуется: отменённый job был
  `Pending` и просто перестаёт быть кандидатом выбора.

### Четвёртый срез: остановка звучащего (задача 019, реализован)

Решения четвёртого среза:

* на очереди `announcement_playback` появился третий тип сообщения тем же
  способом, что и второй (уровни 1 и 3 ADR 002): дискриминатор
  `announcement_playback.stop`, ещё одна ветка `match` в уже существующем
  сериализаторе — реестр decoders по-прежнему не понадобился;
* владельцем перехода `Playing -> Interrupted` остаётся агрегат `PlaybackJob`.
  Переход выполняется только по факту `audio.playback_interrupted` от агента, а
  не в момент приёма HTTP-команды диспетчера: транспорт асинхронный, поэтому до
  подтверждения агентом job всё ещё физически звучит;
* `StopAnnouncementPlayback` — нейтральная операторская команда, а не доменное
  событие: core её публикует по прямому HTTP-запросу диспетчера
  (`POST /api/v1/dispatcher/playback-queue/{announcementId}/stop`), без
  изменения агрегата `Announcement`;
* playback обрабатывает команду идемпотентно через флаг `stopRequested` на
  `PlaybackJob`: только совпадающий `Playing`-job без уже выставленного флага
  получает от playback `audio.stop`; неизвестный, ожидающий или терминальный job
  подтверждается без эффекта, повторная доставка не шлёт второй `audio.stop`;
* контракт Playback → Agent (`agent_commands`): `audio.stop` с `messageId`,
  `jobId`, `occurredAt`, `schemaVersion: 1`. У агента нет `announcementId`,
  приоритетов или иной аэропортовой модели — только `jobId`;
* чтобы stop не встал за долгим `audio.play_sequence`, агент подтверждает play
  сразу после запуска фонового воспроизведения (не после завершения) и хранит
  один активный `jobId` с `CancellationTokenSource`; второй `play` не
  запускается, пока активный job не сообщил терминальный результат. Слот
  активного job освобождается до публикации терминального события — иначе
  следующий `play`, отправленный playback сразу по этому событию, не находит
  агент свободным;
* отмена процесса плеера не является ошибкой: `SequencePlayer` пробрасывает
  `OperationCanceledException`, Worker публикует `audio.playback_interrupted`
  вместо `audio.playback_failed`. Реальная реализация убивает дочерний процесс
  плеера целиком (process tree), simulate-режим корректно реагирует на
  cancellation token;
* при первом переходе публикуется обратное событие
  `AnnouncementPlaybackInterrupted` (`event: announcement_playback.interrupted`,
  контракт как у остальных) и запускается следующий `Pending`-job; повторный
  `interrupted` идемпотентен и не продвигает очередь второй раз;
* core-приём receipt не меняется (generic по имени события); read-модель
  `ListPlaybackQueue` добавляет терминальный `state: interrupted` — строка
  уходит в «Недавние» как «Прервано». Web показывает кнопку «Стоп» только у
  играющей строки, с подтверждением и polling-инвалидацией.

### Пятый срез: повтор continuation-задания (задача 020, реализован)

Владелец повтора — playback, а не core: интервал — вопрос «когда воспроизвести»,
он принадлежит playback (правило #2 не позволяет playback знать про рейсы, но и
не обязывает core диктовать таймер). За всю серию core публикует ровно два
сообщения: `RequestAnnouncementPlayback` с `repeatRule` на старте и
`StopAnnouncementRepeat` на конце; playback самостоятельно крутит цикл между ними.

Решения пятого среза:

* повторяемый job рождается не в `Pending`, а сразу в `Scheduled`, с
  `scheduledAt = createdAt + everyMinutes` — первый тик звучит через один
  интервал после открытия регистрации, а не одновременно с одноразовым
  `check_in_opening`;
* завершение (`complete()`) для повторяемого job без флага остановки не
  терминирует, а **re-arm**: `Playing -> Scheduled`,
  `scheduledAt = completedAt + everyMinutes`. Каденция completion-relative:
  следующий тик отсчитывается от фактического завершения предыдущего, поэтому
  долгая очередь перед тиком сдвигает ритм без «догона» пропущенных тиков.
  `fail()`/`interrupt()` остаются терминальными и для повторяемого job без
  исключения — сбой или чужая остановка не «лечат» повтор попыткой снова;
* `Scheduled` не является кандидатом отбора очереди (аудиовыход свободен для
  других заданий между тиками). Будить job в момент `scheduledAt` некому «извне»,
  поэтому playback планирует собственную delayed-команду
  `WakeScheduledPlaybackJob` (`DelayStamp` на выделенном транспорте `scheduler`,
  своя очередь и DSN) при рождении job и на каждый re-arm; таймер целиком внутри
  playback, cron не вводится. Приём delayed-команды переводит `Scheduled ->
  Pending`, дальше — обычный приоритетный отбор; если job уже терминален (успел
  прийти стоп), wake — no-op;
* остановка серии — идемпотентный флаг `repeatCancelled` по образцу
  `stopRequested` (019): `StopAnnouncementRepeat` находит job по `announcementId`
  под write lock. Флаг **не обрывает** играющий тик — тот доигрывает и
  терминируется на своём следующем `complete()`; но job, застигнутый между
  тиками (`Scheduled`/`Pending`), терминируется немедленно
  (`Scheduled|Pending -> Completed`), не дожидаясь следующего wake;
* `StopAnnouncementRepeat` — четвёртый тип сообщения на очереди
  `announcement_playback`, тем же способом, что второй и третий (уровни 1 и 3
  ADR 002): дискриминатор `command: announcement_playback.stop_repeat`, ещё одна
  ветка `match` в уже существующем сериализаторе;
* приёмная сторона core не меняется: терминация между тиками публикует уже
  существующее `AnnouncementPlaybackCompleted` (job действительно ушёл в
  `Completed`), а re-arm никакого события не публикует — с точки зрения receipts
  повторяющийся job выглядит как один длинный `started` без промежуточных
  `completed` на каждый тик. Это осознанный компромисс: `ListPlaybackQueue`
  использует существующий generic-механизм без специального кода для повтора, но
  экран «Статус» в результате показывает continuation играющей непрерывно, пока
  открыта регистрация, а не только в момент реального тика — известное
  ограничение, не исправленное этим срезом (см. открытые решения).

Состояние:

* `id`;
* тип;
* приоритет;
* статус;
* `scheduledAt`;
* аудиопоследовательность;
* правило повтора;
* `createdAt`.

Статусы:

```text
Pending
Scheduled
Playing
Completed
Interrupted
Cancelled
Failed
```

Инварианты:

* одновременно выполняется не более одного задания на один audio output;
* завершённое или отменённое задание нельзя запустить повторно;
* экстренное задание имеет приоритет `1000`;
* рейсовое объявление имеет приоритет `100`;
* дополнительное объявление имеет приоритет `50`;
* фоновая музыка имеет приоритет `0`;
* emergency mode приостанавливает обычную очередь;
* после завершения emergency mode обычная очередь может продолжиться;
* повторно доставленная команда не должна создавать дубликат задания.

Playback принимает integration commands:

* `RequestAnnouncementPlayback`;
* `CancelAnnouncementPlayback`;
* `StopAnnouncementPlayback` (задача 019);
* `StopAnnouncementRepeat` (задача 020);
* `ActivateEmergencyMode`;
* `DeactivateEmergencyMode`.

Playback публикует integration events:

* `AnnouncementPlaybackQueued`;
* `AnnouncementPlaybackStarted`;
* `AnnouncementPlaybackCompleted`;
* `AnnouncementPlaybackCancelled`;
* `AnnouncementPlaybackInterrupted` (задача 019);
* `EmergencyModeActivated`;
* `EmergencyModeDeactivated`.

Повтор (задача 020) не добавляет новый тип события: re-arm ничего не публикует, а
терминация серии между тиками переиспользует `AnnouncementPlaybackCompleted`.

# Domain events и integration messages

Domain event существует внутри bounded context и описывает бизнес-факт.

Integration message пересекает границу сервиса и является публичным контрактом.
Внутренняя доменная сущность не должна сериализоваться и отправляться напрямую.

Пример:

```text
dispatcher action on FlightOccurrence
    -> FlightOccurrence transition + create Announcement   одна локальная транзакция
    -> CheckInOpened / AnnouncementCreated                 domain events (факты)
    -> RequestAnnouncementPlayback                         integration command (async)
```

Domain events перехода описывают уже состоявшийся факт и не запускают создание
основного объявления. `RequestAnnouncementPlayback` пересекает границу сервиса
после коммита локальной транзакции.

Требования к integration messages:

* содержат стабильный `messageId`;
* содержат идентификатор корреляции, если участвуют в общем сценарии;
* содержат только необходимые скалярные данные и DTO;
* имеют версионируемый контракт;
* допускают повторную доставку;
* не требуют общей базы данных между сервисами.

# Согласованность и транзакции

Сильная согласованность требуется только внутри одного агрегата и одной локальной
транзакции.

Исключение в рамках одного сервиса: запуск рейсового объявления меняет
`FlightOccurrence` и создаёт `Announcement` в **одной локальной транзакции** Core.
Это два агрегата двух контекстов, но в модульном монолите с общей БД они
выражают один неразделимый бизнес-факт, поэтому атомарность здесь — осознанный
прагматичный выбор, а не сага. Eventual consistency и outbox применяются только
на настоящей сервисной границе (Core → `aeroflow-playback`).

Между bounded contexts предпочтительно используется eventual consistency.
Синхронный application-port допустим как явно обоснованное исключение, когда
текущий use case требует немедленного ответа другого контекста. Такой вызов не
создаёт общей транзакционной границы между агрегатами.

Для надёжной публикации integration messages из транзакции Core или Playback
следует использовать transactional outbox. Решение по первой версии: durable
outbox **отложен**, транспорт между сервисами — RabbitMQ, публикация выполняется
из application layer после коммита (post-commit publisher). Это осознанный
временный компромисс с известным риском потери сообщения при крахе между коммитом
и публикацией. До внедрения durable outbox место публикации сообщений остаётся в
application layer, чтобы и outbox, и транспорт можно было ввести без изменения
доменной модели.

Optimistic locking следует добавлять агрегатам с конкурентными командами, прежде
всего `FlightOccurrence` (запуск объявления грузит occurrence на запись) и
будущему `PlaybackJob`. Для каталога
`FlightDefinition` это можно сделать при появлении реальных конфликтов редактирования.

# Идемпотентность

Идемпотентность обязательна для команд, приходящих через Messenger/RabbitMQ.

Минимальная стратегия:

* каждое integration message имеет `messageId`;
* получатель хранит обработанные идентификаторы или использует уникальный
  бизнес-ключ;
* повторная обработка возвращает уже достигнутый результат;
* операции перехода состояния проверяют текущее состояние агрегата.

HTTP-команды, которые могут быть повторены клиентом, при необходимости используют
idempotency key.

Value Object вводится, если выполняется хотя бы одно условие:

* значение имеет собственную валидацию;
* значение сравнивается по смыслу;
* значение участвует в нескольких агрегатах;
* использование примитива регулярно приводит к ошибкам;
* объект заметно улучшает ubiquitous language.

Если значение только переносится из API в базу и не имеет поведения, примитив
допустим.

# Репозитории

Репозиторий предоставляет операции, нужные use case, а не копирует весь API ORM.

Правила:

* интерфейс располагается рядом с доменной моделью;
* реализация располагается в Infrastructure;
* наружу возвращаются доменные объекты;
* методы называются в языке модели;
* `save()` сохраняет aggregate root;
* сложные read-only экраны могут использовать отдельные query services и DTO,
  минуя загрузку агрегатов.

CQRS применяется локально и прагматично: command side защищает инварианты, query
side оптимизируется под чтение. Отдельные сервисы или базы для CQRS не требуются
без измеримой необходимости.

# Тестирование доменной модели

Приоритет тестов:

1. Инварианты и переходы состояния aggregate root.
2. Application handlers и публикуемые события.
3. Репозитории и ORM mapping с PostgreSQL.
4. Публичные HTTP и message contracts.
5. Небольшое число end-to-end сценариев.

Domain tests не должны загружать Symfony kernel или базу данных.

Основная форма теста:

```text
Given: состояние агрегата
When: бизнес-действие
Then: новое состояние, инварианты и domain events
```

Не следует тестировать private methods или дублировать тестами поведение Symfony,
Doctrine и других библиотек.

# Ubiquitous Language

* **Flight Definition** — многократно используемая карточка рейса.
* **Flight Schedule** — будущее правило времени и повторяемости.
* **Flight Occurrence** — будущий конкретный вылет в определённую дату.
* **Check-in** — процесс регистрации пассажиров на вылет.
* **Boarding** — процесс посадки через назначенный выход.
* **Announcement** — бизнес-сообщение, которое должно быть озвучено.
* **Audio Asset** — зарегистрированный аудиоматериал.
* **Audio Sequence** — упорядоченный набор audio assets для одного объявления.
* **Playback Job** — техническое задание на воспроизведение audio sequence.
* **Emergency Mode** — режим, в котором экстренное воспроизведение приоритетнее
  обычной очереди.
* **Domain Event** — факт внутри bounded context.
* **Integration Command** — запрос другому bounded context выполнить действие.
* **Integration Event** — публичное сообщение о результате или изменении.

# Открытые решения

Принятые решения (зафиксированы для occurrence-driven запуска объявлений):

* `FlightOccurrence` вводится до расписания как core-срез operational рейса;
  ручной диспетчерский поток идёт `FlightDefinition (шаблон) -> FlightOccurrence
  -> Announcement`;
* lifecycle вылета и прилёта зафиксирован в разделе `FlightOccurrence`;
* запуск рейсового объявления оркеструется в `Flight Operations\Application`;
  `Announcements` создаёт объявление через порт, принадлежащий `Flight
  Operations`, и не мутирует occurrence;
* переход occurrence и создание `Announcement` атомарны в одной локальной
  транзакции Core; готовность объявления — предусловие перехода;
* форма API — action-эндпоинты над `FlightOccurrence` (см. задачу 011);
  definition-based путь создания рейсового объявления деприкейтится;
* интеграция Core → Playback (см. задачу 013): `RequestAnnouncementPlayback`
  публикует контекст `Announcements` после коммита, реагируя на `AnnouncementCreated`;
  транспорт — RabbitMQ; durable transactional outbox отложен (post-commit publisher
  как временный компромисс с известным риском); первый срез playback создаёт
  `PlaybackJob` идемпотентно по `messageId`, без очереди и emergency mode.
* ручной поток допускает **несколько occurrence по одной карточке в день** (см.
  задачу 014): повторный прогон создаётся через `StartNextManualFlightOccurrence`
  с серверным инкрементом `sequenceNumber`; защита от дубля — доменное состояние
  (новый прогон только когда последний достиг финального статуса цикла —
  `boarding`/`arrival_announced` или терминальных `completed`/`cancelled`), а не
  клиентский idempotency key. `sequenceNumber` остаётся синтетическим ключом
  различения прогонов; при
  появлении `FlightSchedule` идентичность occurrence пересматривается в пользу
  времени вылета (`operationalDateTime`), а синтетический счётчик может быть снят.
* дискриминация входящих integration messages — ADR 002
  (`adr/002-inbound-message-discrimination.md`): дискриминатор типа — поле в теле
  сообщения (`command` для команд, `event` для событий), вводится при появлении
  второго типа на одной очереди — введён задачей 018 (срез
  `CancelAnnouncementPlayback`) и подтверждён задачей 019 (срез
  `StopAnnouncementPlayback`, третий тип на той же очереди тем же `match`);
  emergency-команды получают отдельный транспорт; registry декодеров заранее не
  вводится.
* остановка звучащего объявления (см. задачу 019): `StopAnnouncementPlayback` —
  нейтральная операторская команда, не доменное событие и не изменение
  `Announcement`; переход `PlaybackJob` `Playing -> Interrupted` происходит
  только по факту `audio.playback_interrupted` от агента, никогда по самой
  HTTP-команде диспетчера.
* правила повторов объявления «продолжение регистрации» (см. задачу 020,
  закрывает открытый вопрос ниже): повтор владеет **playback**, а не core —
  один и тот же `Announcement` циклится вокруг `Scheduled -> Pending -> Playing
  -> Scheduled`, а не per-tick создание нового объявления через
  `continueCheckIn()`/`CheckInContinued` (эти метод и событие помечены
  `@deprecated`, целевая модель из более раннего варианта документа не
  реализуется). Интервал приходит один раз в `RequestAnnouncementPlayback.repeatRule`
  при старте серии; конец серии — отдельная нейтральная команда
  `StopAnnouncementRepeat`, четвёртый тип на очереди `announcement_playback`
  (ADR 002, тот же `match`). Открытая read-модель `ListPlaybackQueue` не получила
  отдельного кода под повтор: continuation отображается играющей непрерывно, пока
  идёт серия, а не только в момент реального тика — принятое ограничение
  тонкого среза, а не свежий баг (см. «Пятый срез» `PlaybackJob`).
* генерация объявлений через TTS (закрывает открытый вопрос
  `architecture.md` «Будет ли TTS в первой версии или позже?»): синтез —
  отдельный stateless-сервис `aeroflow-tts`, нейтральный к домену
  (`text/language/voice -> wav`), в роли Audio Catalog, **не** новый bounded
  context; Core зовёт его синхронно через порт `TextToSpeechPort` как пре-шаг при
  сохранении текста конфигурации; кэш и идемпотентность по хешу текста — в Core;
  движок и стек сервиса — деталь реализации за HTTP-контрактом (см. раздел
  «Генерация assets через TTS»). Реализация — отдельным таском.

Следующие вопросы должны быть уточнены соответствующим use case до реализации:

* модель задержки, переноса и отмены рейса;
* правила конфликтов Gate и CheckInCounter;
* граница агрегата для AnnouncementTemplate;
* хранится ли готовая audio sequence в Announcement или создаётся отдельная
  неизменяемая версия;
* формат версионирования integration messages (первый контракт несёт `schemaVersion`);
* объём audit log и сроки хранения;
* привести `GateDirectory` и `CheckInCounterDirectory` к стилю command/query
  handlers, как `AudioPrompt`, либо сознательно закрепить за справочными
  агрегатами паттерн тонкой директории — сейчас они одинаковы по форме.

Открытый вопрос не должен решаться случайно внутри controller, Doctrine repository
или message handler. Решение сначала фиксируется в этом документе.
