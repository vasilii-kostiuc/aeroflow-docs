# Manual Announcement Domain Model

Status: Done  
Bounded context: Announcements  
Service: aeroflow-core  
Change type: Domain

## Goal

Создать доменную модель ручного рейсового объявления, которая фиксирует:

* что диспетчер намерен объявить;
* к какой постоянной карточке рейса относится объявление;
* какие операционные параметры нужны для выбранного типа объявления;
* на каких языках и в каком порядке оно должно прозвучать;
* текущее состояние подготовки объявления.

Первый результат ограничен чистой доменной моделью:

```text
dispatcher action
    -> Announcement::create(...)
    -> PendingPreparation
    -> AnnouncementCreated
```

Наличие legacy-аудиофайлов, способ подготовки аудио и транспорт до playback для
этой задачи не требуются.

## Architectural Classification

Изменение относится к bounded context `Announcements` в `aeroflow-core`.

`Announcement` отвечает на вопрос:

```text
Что должно быть объявлено и почему?
```

Он не решает:

* из каких физических файлов собрать объявление;
* когда поставить объявление в очередь;
* как воспроизвести звук.

Затрагиваемый aggregate root:

* `Announcement`.

Связанные агрегаты:

* `FlightDefinition` — только по идентификатору, без ORM-связи и объектного графа.

Domain events:

* `AnnouncementCreated`;
* `AnnouncementLanguagesChanged`;
* `AnnouncementCancelled`.

Integration messages в этой задаче не публикуются.

## Scope

* создать aggregate root `Announcement`;
* создать enum `AnnouncementType`;
* создать enum `AnnouncementStatus`;
* создать Value Objects `GateCode`, `CheckInCounterRange`, `LanguageCode` и
  `AnnouncementLanguages`;
* реализовать создание поддерживаемых ручных рейсовых объявлений;
* реализовать изменение количества и порядка языков до подготовки;
* реализовать отмену объявления до отправки на воспроизведение;
* публиковать domain events агрегата;
* покрыть инварианты и поведение чистыми domain unit tests;
* зафиксировать реализованные решения в `domain-model.md` после завершения.

## Out of Scope

* application commands и handlers;
* проверка существования и активности `FlightDefinition`;
* Doctrine mapping, repository, migration и PostgreSQL;
* HTTP API;
* frontend диспетчера;
* `AudioAsset`, шаблоны, содержимое языковых вариантов и `audioSequence`;
* выбор способа подготовки конкретного языка: готовое аудио или text-to-speech;
* интеграция с TTS-провайдером и хранение сгенерированного аудио;
* подготовка объявления и статус `Prepared`;
* `RequestAnnouncementPlayback` и другие integration messages;
* `aeroflow-playback` и `PlaybackJob`;
* автоматические повторы;
* дополнительное и экстренное объявления;
* `FlightSchedule` и `FlightOccurrence`;
* изменение операционного состояния рейса.

## Domain Model

### Announcement

`Announcement` является aggregate root.

Состояние первого этапа:

* `id` — UUID, создаётся доменом;
* `type` — тип объявления;
* `flightDefinitionId` — UUID постоянной карточки рейса;
* `checkInCounterRange` — диапазон стоек для объявлений о регистрации;
* `gateCode` — выход для приглашения на посадку;
* `languages` — непустой упорядоченный набор языков объявления;
* `status` — состояние подготовки;
* `createdAt`;
* `cancelledAt`.

Поля `checkInCounterRange` и `gateCode` присутствуют только для тех типов, которым
они необходимы. Их допустимые комбинации защищаются фабричными методами агрегата,
а не передаются через произвольный массив параметров.

### AnnouncementType

На первом этапе поддерживаются:

```text
CheckInOpening
CheckInClosing
BoardingInvitation
Arrival
```

`CheckInContinuation` не входит в задачу, потому что является автоматическим
повтором и требует правил планирования.

`Additional` и `Emergency` будут добавлены отдельными сценариями после определения
модели их содержимого.

### AnnouncementStatus

На первом этапе поддерживаются:

```text
PendingPreparation
Cancelled
```

Новое объявление всегда создаётся в `PendingPreparation`.

Статусы `Prepared` и `PlaybackRequested` не добавляются заранее. Их переходы и
инварианты будут определены вместе с реальным сценарием подготовки аудио и
интеграцией с playback.

### Public Behavior

Предполагаемый публичный API агрегата:

```php
Announcement::openCheckIn(
    string $flightDefinitionId,
    CheckInCounterRange $counterRange,
    AnnouncementLanguages $languages,
): Announcement;

Announcement::closeCheckIn(
    string $flightDefinitionId,
    CheckInCounterRange $counterRange,
    AnnouncementLanguages $languages,
): Announcement;

Announcement::inviteToBoard(
    string $flightDefinitionId,
    GateCode $gateCode,
    AnnouncementLanguages $languages,
): Announcement;

Announcement::announceArrival(
    string $flightDefinitionId,
    AnnouncementLanguages $languages,
): Announcement;

$announcement->cancel();
$announcement->changeLanguages($languages);
$announcement->pullEvents();
```

UUID может быть представлен существующим проектным подходом, если это не создаёт
зависимость домена от application layer.

## Business Rules

* идентификатор объявления создаётся до публикации `AnnouncementCreated`;
* `flightDefinitionId` обязателен и должен иметь корректный UUID-формат;
* проверка существования и активности карточки относится к будущему application
  handler, поскольку `FlightDefinition` является другим агрегатом;
* начало и окончание регистрации требуют корректный диапазон стоек;
* приглашение на посадку требует корректный код выхода;
* прибытие не принимает стойки регистрации или выход;
* объявление содержит хотя бы один язык;
* один язык не может повторяться в объявлении;
* порядок языков является значимым и определяет будущий порядок подготовки и
  воспроизведения языковых вариантов;
* до подготовки можно добавлять, удалять и переупорядочивать языки заменой
  `AnnouncementLanguages`;
* повторная установка того же набора в том же порядке идемпотентна;
* после отмены языки изменять нельзя;
* новое объявление получает статус `PendingPreparation`;
* отменить можно только объявление, которое ещё ожидает подготовки;
* повторная отмена идемпотентна и не публикует повторное событие;
* отменённое объявление нельзя вернуть в активное состояние;
* агрегат не хранит `FlightDefinition` как объект;
* агрегат не содержит пути к файлам или сведения о playback;
* на этапе создания агрегат не решает, будет конкретный язык подготовлен из
  существующего аудио или из текста через TTS.

## Value Objects

### GateCode

`GateCode`:

* не может быть пустым;
* нормализуется в верхний регистр;
* допускает короткое буквенно-цифровое обозначение выхода;
* имеет явное строковое представление и сравнение по значению.

Точную регулярную форму следует зафиксировать тестами реализации. Она не должна
быть уже реальных обозначений аэропорта без подтверждённого бизнес-требования.

### CheckInCounterRange

`CheckInCounterRange` представляет одну стойку или непрерывный диапазон:

```text
5
5-8
```

Инварианты:

* номера стоек являются положительными целыми числами;
* начальная стойка не больше конечной;
* одна стойка представляется диапазоном с одинаковыми началом и концом;
* сравнение выполняется по значениям начала и конца;
* форматирование является ответственностью Value Object.

### LanguageCode

`LanguageCode` представляет язык объявления:

* не может быть пустым;
* использует BCP 47-совместимое обозначение, например `ro`, `ru`, `en` или
  `ro-MD`;
* нормализует регистр языкового кода и регионального подтага;
* сравнивается по нормализованному значению.

Поддерживаемые системой языки не зашиваются в enum. Их доступность будет
проверяться будущим application use case или справочником языков.

### AnnouncementLanguages

`AnnouncementLanguages` является непустой упорядоченной коллекцией
`LanguageCode`.

Инварианты:

* содержит хотя бы один язык;
* не содержит дубликатов;
* сохраняет порядок, выбранный диспетчером или заданный конфигурацией;
* предоставляет языки как значения, не раскрывая изменяемую внутреннюю коллекцию.

## Future Language Variants

Способ подготовки является свойством конкретного языкового варианта, а не всего
объявления. В дальнейшем одно объявление сможет сочетать разные источники:

```text
Announcement
├── ro -> recorded audio
├── ru -> recorded audio
└── en -> text -> TTS -> generated audio
```

Будущая модель подготовки должна различать как минимум:

* вариант из существующих `AudioAsset`;
* текстовый вариант, предназначенный для text-to-speech;
* итоговый подготовленный audio sequence, пригодный для playback.

Текст и ссылки на аудио не добавляются в текущую модель заранее. Их структура,
жизненный цикл и принадлежность агрегату будут определены задачей подготовки
многоязычного объявления.

Speech-to-text (STT) преобразует речь в текст и не относится к описанному
сценарию. Для озвучивания текста используется text-to-speech (TTS).

## Domain Events

### AnnouncementCreated

Содержит только данные, необходимые внутренним подписчикам:

* `announcementId`;
* `type`;
* `flightDefinitionId`;
* `languages`;
* `occurredAt`.

Операционные параметры остаются состоянием агрегата и не добавляются в событие
без конкретного подписчика.

### AnnouncementLanguagesChanged

Содержит:

* `announcementId`;
* новый упорядоченный набор `languages`;
* `occurredAt`.

### AnnouncementCancelled

Содержит:

* `announcementId`;
* `occurredAt`.

События являются внутренними для `Announcements` и не являются публичным
контрактом между сервисами.

## Test Scenarios

* создаётся объявление о начале регистрации с диапазоном стоек;
* создаётся объявление об окончании регистрации с диапазоном стоек;
* создаётся приглашение на посадку с выходом;
* создаётся объявление о прибытии без дополнительных параметров;
* объявление создаётся с одним языком;
* объявление сохраняет порядок нескольких языков;
* количество языков может увеличиваться и уменьшаться;
* языки можно переупорядочить до подготовки;
* повторная установка того же порядка не публикует событие;
* языки отменённого объявления изменить нельзя;
* пустой набор языков отклоняется;
* повторяющийся язык отклоняется;
* корректные BCP 47-совместимые коды нормализуются;
* некорректный `LanguageCode` отклоняется;
* каждое новое объявление имеет UUID и статус `PendingPreparation`;
* создание публикует ровно один `AnnouncementCreated`;
* `AnnouncementCreated` содержит уже созданный идентификатор агрегата;
* некорректный UUID карточки рейса отклоняется;
* пустой или некорректный `GateCode` отклоняется;
* некорректный диапазон стоек отклоняется;
* объявление в `PendingPreparation` отменяется и публикует
  `AnnouncementCancelled`;
* повторная отмена не меняет состояние и не публикует событие;
* фабричные методы не позволяют создать несовместимую комбинацию типа и
  параметров;
* domain tests не загружают Symfony kernel и базу данных.

## Definition of Done

* доменная модель находится в `src/Announcements/Domain`;
* `Announcement` наследует существующий `AggregateRoot`;
* публичные фабричные методы выражают бизнес-действия, а не техническое CRUD
  `create(type, parameters)`;
* отсутствуют публичные setters;
* отсутствуют зависимости от Doctrine repository, Messenger, HTTP и playback;
* все перечисленные domain tests проходят;
* `domain-model.md` обновлён и описывает фактически реализованное состояние
  bounded context `Announcements`;
* `composer test` и существующие проверки проекта проходят.
