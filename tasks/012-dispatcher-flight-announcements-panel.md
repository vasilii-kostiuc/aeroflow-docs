# Dispatcher Flight Announcements Panel

Status: Draft  
Bounded context: Announcements, Flight Operations, Localization  
Service: aeroflow-web, aeroflow-core  
Change type: Frontend, API

## Goal

Сделать первый рабочий интерфейс диспетчера для запуска рейсовых объявлений на
основе активных `FlightOccurrence`.

Административный интерфейс уже отвечает за настройку рейсов и конфигураций
объявлений. Новый экран должен быть легче и ближе к старой системе: это единая
рабочая панель, где тип объявления выбирается кнопочным фильтром, а список
рейсов и форма запуска перестраиваются внутри того же экрана.

Экран зависит от core-задачи `011-flight-occurrences-core.md`: dispatcher UI
работает с конкретными operational рейсами, а не хранит состояние на
`FlightDefinition`.

Основной сценарий:

```text
Диспетчер открывает рабочую панель
  ↓
Выбирает режим объявления: начало регистрации, продолжение/окончание
регистрации, посадка или прибытие
  ↓
Видит список подходящих рейсов
  ↓
Выбирает рейс
  ↓
Уточняет только необходимые операционные параметры
  ↓
Запускает объявление
  ↓
Система создаёт Announcement
```

Если подходящий `FlightOccurrence` ещё не создан, core может создать ручной
occurrence из активного `FlightDefinition` до запуска первого объявления.

## Architectural Classification

Задача затрагивает:

* `Flight Operations` — чтение и изменение `FlightOccurrence` для
  диспетчерского сценария;
* `Announcements` — создание рейсового `Announcement` по выбранному типу и
  операционным параметрам;
* `Localization` — выбор языков объявления из справочника языков;
* `Audio Catalog` — только косвенно, через готовность конфигурации и
  segment-based templates.

Изменение в основном frontend/application. Доменный срез `FlightOccurrence`
реализуется отдельной core-задачей `011-flight-occurrences-core.md`.

Затрагиваемые агрегаты:

* `FlightOccurrence` — выбранный operational рейс и источник текущего статуса;
* `FlightDefinition` — используется как шаблон/карточка для создания manual
  occurrence и отображения постоянных данных рейса;
* `FlightAnnouncementConfig` — используется как источник готовности и доступных
  вариантов;
* `Announcement` — создаётся при запуске диспетчерского действия.

При создании объявления используется уже существующее событие
`AnnouncementCreated`. Для изменения операционного состояния `FlightOccurrence`
нужны события `CheckInOpened`, `CheckInContinued`, `CheckInClosed`,
`BoardingStarted`, `ArrivalAnnounced` или их ближайшие существующие аналоги.

`aeroflow-playback` и `aeroflow-agent` не затрагиваются в этой задаче. Если
создание `Announcement` уже публикует integration command в playback на момент
реализации, dispatcher UI использует существующее поведение без отдельной
логики воспроизведения.

## Product Principle

Панель диспетчера не должна дублировать админку.

Она должна отвечать на вопросы:

```text
Какое объявление диспетчер хочет запустить сейчас?
Какие рейсы подходят для выбранного объявления?
Какие минимальные данные нужно уточнить перед запуском?
```

Она не должна отвечать на вопросы:

```text
Как устроен шаблон?
Какие сегменты входят в вариант?
Как настроить аудиоассеты?
```

## User Experience

Экран должен быть компактным и пригодным для повторяющейся работы.

### Action-First Workflow

Основной UX должен повторять привычный сценарий старой системы, но оставаться
единым рабочим пространством:

```text
Кнопочные фильтры действий сверху:
  Начало регистрации
  Продолжение регистрации
  Окончание регистрации
  Посадка
  Прибытие

Общий список рейсов ниже:
  показывает только рейсы, подходящие под выбранный фильтр
  сохраняет поиск и быстрый контекст диспетчера

Панель запуска:
  показывает выбранный рейс
  показывает короткую форму параметров текущего действия
```

Это не должны быть отдельные страницы или разрозненные списки. Диспетчер должен
быстро переключаться между вариантами объявлений и оставаться в одном интерфейсе.

Примеры фильтрации:

* `Начало регистрации` — активные рейсы, для которых регистрация ещё не открыта;
* `Продолжение регистрации` — рейсы, у которых регистрация уже открыта;
* `Окончание регистрации` — рейсы, у которых регистрация открыта или идёт
  продолжение регистрации;
* `Посадка` — рейсы, у которых регистрация завершена и ещё не запущена посадка;
* `Прибытие` — активные прилётные рейсы, по которым ещё можно объявить
  прибытие.

Статус не должен выводиться только из `FlightAnnouncementConfig`: это настройка,
а не факт запуска. Статус также не должен жить в playback, потому что playback
не знает бизнес-логику рейсов.

Статус принадлежит `FlightOccurrence` в `Flight Operations`. Это позволяет
использовать одинаковый механизм для:

* ручного диспетчерского запуска;
* будущего создания occurrence по расписанию;
* будущих автоматических или полуавтоматических объявлений.

## Occurrence State

Dispatcher UI использует lifecycle `FlightOccurrence`:

```text
scheduled
  -> check_in_open
  -> check_in_closed
  -> boarding
  -> completed
```

Для прилёта допустим отдельный простой lifecycle:

```text
scheduled
  -> arrival_announced
  -> completed
```

Для ручного режима dispatcher UI не создаёт отдельную модель состояния. Он
получает `operationStatus` из `FlightOccurrence` и передаёт
`flightOccurrenceId` при запуске объявления.

Создание `Announcement` и изменение `FlightOccurrence` должны выполняться в
одном application use case или в согласованной транзакционной
последовательности:

```text
validate transition
  -> create Announcement
  -> update FlightOccurrence status/snapshot
  -> publish AnnouncementCreated and state transition event
```

Нельзя сначала создать объявление, а потом оставить `FlightOccurrence` в прежнем
статусе при ошибке сохранения состояния. Повторная отправка одного и того же
запроса не должна ломать состояние; если потребуется защита от двойного клика,
добавить idempotency key на уровне API.

### Layout

Рекомендуемая структура:

```text
Верхняя панель:
  кнопочный фильтр типов объявления
  текущий выбранный тип

Основная область:
  только подходящие рейсы
  поиск по номеру рейса, направлению, авиакомпании
  компактные карточки или список рейсов

Панель запуска справа или снизу:
  выбранный рейс
  выбранное действие
  минимальная форма параметров
  кнопка запуска

Служебная зона:
  последние созданные объявления по выбранному рейсу
  статус результата последнего запуска
```

Для desktop допустим split layout: список рейсов слева, панель запуска справа.
Для mobile/tablet кнопочный фильтр остаётся сверху, затем список, затем форма
выбранного рейса. Переключение типа объявления не должно уводить пользователя на
другую страницу.

### Flight Card

В карточке рейса показывать только диспетчерски полезную информацию:

* номер рейса;
* направление или аэропорт назначения/прибытия;
* авиакомпания, если доступна;
* тип направления: departure/arrival, если доступен;
* краткий индикатор готовности объявлений.

Не показывать технические поля, UUID, segment details и admin-only controls.

### Announcement Actions

Показать действия как кнопочный фильтр единой панели:

* открыть регистрацию;
* продолжить регистрацию;
* закрыть регистрацию;
* пригласить на посадку;
* объявить прибытие.

Внутри выбранного действия показывать только подходящие рейсы. Рейс попадает в
список только если:

* occurrence активен и не завершён;
* текущий рабочий статус допускает выбранное действие;
* для типа объявления есть активная конфигурация;
* есть минимум один активный языковой вариант;
* конфигурация может быть разрешена в `AudioSequence`.

Если рейс не попал в список, он не должен мешать рабочему сценарию. Для
диагностики можно добавить переключатель "показать недоступные", где UI
показывает короткую причину:

```text
Регистрация уже открыта
Регистрация ещё не открыта
Посадка уже объявлена
Нет активного варианта на выбранном языке
Не настроены аудиофрагменты для выхода A5
Конфигурация объявления выключена
```

Причина должна быть пригодной для диспетчера, без внутренних exception names.

### Minimal Parameters

Общие параметры:

* `languages` — `MultiSelect` из read-only списка языков;
* порядок языков сохраняется в payload;
* по умолчанию выбирать первый активный язык или заранее выбранный набор
  диспетчерского профиля, если такой появится позже.

Для регистрационных объявлений:

* выбор одной или нескольких стоек регистрации;
* стойки берутся из активного справочника `CheckInCounter`;
* диапазон стоек не используется, если API уже перешёл на `checkInCounterIds`.

Для приглашения на посадку:

* выбор одного выхода;
* выход берётся из активного справочника `Gate`.

Для прибытия:

* дополнительных параметров нет.

## Backend Scope

Использовать существующие endpoints, если они уже покрывают сценарий:

```text
GET  /api/v1/dispatcher/flight-occurrences
POST /api/v1/flight-occurrences/{id}/check-in:open
POST /api/v1/flight-occurrences/{id}/check-in:close
POST /api/v1/flight-occurrences/{id}/boarding
POST /api/v1/flight-occurrences/{id}/arrival
GET  /api/v1/languages
GET  /api/v1/admin/check-in-counters?active=true
GET  /api/v1/admin/gates?active=true
```

Запуск объявления — это action над `FlightOccurrence` (см. задачу 011), а не
`POST /api/v1/announcements`.

Read-only dispatcher API не должен дублировать доменную логику:

```http
GET /api/v1/dispatcher/flight-occurrences
```

Response должен быть read model для панели:

```json
{
  "success": true,
  "data": [
    {
      "id": "uuid",
      "flightDefinitionId": "uuid",
      "flightNumber": "5F 123",
      "direction": "departure",
      "airportCode": "IST",
      "airportName": "Istanbul",
      "active": true,
      "operationStatus": "scheduled",
      "announcementReadiness": [
        {
          "type": "check_in_opening",
          "ready": true,
          "eligible": true,
          "languages": ["ro-MD", "ru"],
          "unavailableReason": null
        }
      ]
    }
  ]
}
```

Read model может агрегировать данные из `FlightOccurrence`, `FlightDefinition` и
`FlightAnnouncementConfig`, но не должен менять доменные инварианты.

### Backend Requirements

* dispatcher list возвращает только активные `FlightOccurrence`;
* dispatcher read model поддерживает фильтр по типу действия:
  `announcementType=check_in_opening|check_in_continuation|check_in_closing|boarding_invitation|arrival`;
* при фильтре по действию backend возвращает только рейсы, подходящие по
  рабочему статусу и готовности конфигурации;
* после успешного создания объявления read model отражает новый статус
  occurrence, чтобы рейс перемещался между режимами без ручной правки карточки;
* недопустимый переход состояния отклоняется до создания `Announcement`;
* если backend возвращает недоступные рейсы для диагностики, он должен отдавать
  человекочитаемую причину недоступности;
* список сортируется для рабочего использования: по номеру рейса или текущему
  существующему порядку, если порядок уже определён в API;
* readiness считается на стороне backend, если frontend не может надёжно
  определить её без знания доменных правил;
* создание объявления продолжает идти через application handler `Announcements`,
  но command принимает `flightOccurrenceId` или явно разрешает occurrence до
  создания объявления;
* ошибки создания объявления возвращаются как валидационные ошибки, пригодные
  для отображения рядом с действием.

## Frontend Scope

Добавить dispatcher feature в `aeroflow-web`.

Предлагаемая структура:

```text
src/pages/dispatcher/DispatcherPage.tsx
src/features/dispatcher/api/dispatcherApi.ts
src/features/dispatcher/hooks/useDispatcherFlightOccurrences.ts
src/features/dispatcher/ui/DispatcherOccurrenceList.tsx
src/features/dispatcher/ui/DispatcherActionPanel.tsx
src/features/dispatcher/ui/DispatcherLaunchForm.tsx
```

Если в проекте уже есть другой naming convention, следовать ему.

### Routing

Добавить route для диспетчера:

```text
/dispatcher
```

Route должен быть доступен авторизованным пользователям. Если роли уже
разделены, экран доступен ролям диспетчера и администратора.

### Launch Flow

Frontend создаёт `Announcement` через существующий API.

Payload для segment-based модели:

```json
{
  "type": "check_in_opening",
  "flightOccurrenceId": "uuid",
  "languages": ["ro-MD", "ru"],
  "checkInCounterIds": ["uuid-1", "uuid-3"],
  "gateId": null
}
```

Для `boarding_invitation`:

```json
{
  "type": "boarding_invitation",
  "flightOccurrenceId": "uuid",
  "languages": ["ro-MD"],
  "checkInCounterIds": [],
  "gateId": "uuid"
}
```

Для `arrival`:

```json
{
  "type": "arrival",
  "flightOccurrenceId": "uuid",
  "languages": ["ro-MD"]
}
```

Если backend ещё использует старые поля `checkInCounterStart`,
`checkInCounterEnd` или `gateCode`, необходимо зафиксировать расхождение с
актуальной документацией task `009` и привести API к документации перед или в
рамках реализации.

### UI States

Обязательные состояния:

* загрузка списка рейсов;
* пустой список активных рейсов;
* выбранный режим без подходящих рейсов;
* недоступное действие с причиной;
* валидационная ошибка формы;
* успешное создание объявления;
* ошибка сети или 5xx без потери введённых параметров.

### Design Requirements

* избегать больших административных таблиц;
* использовать плотные карточки или список для рейсов;
* primary action должен быть один на выбранный тип объявления;
* опасные или экстренные сценарии не смешивать с рейсовыми действиями;
* не показывать editor controls для template segments;
* не использовать marketing-style hero layout.

## Out of Scope

* `FlightSchedule`;
* автоматическое определение операционного состояния по расписанию, времени или
  внешним системам;
* queue/playback control screen;
* экстренные и дополнительные объявления;
* настройка шаблонов и аудиоассетов;
* создание или редактирование рейсов;
* аудит действий диспетчера, если audit context ещё не реализован;
* realtime обновления через WebSocket;
* интеграция с табло расписания.

## Acceptance Criteria

Backend:

* есть read model или переиспользованы существующие endpoints для получения
  активных рейсов диспетчера;
* readiness действий не требует от frontend знания внутренней структуры
  segment templates;
* `POST /announcements` создаёт объявление из dispatcher payload и обновляет
  статус `FlightOccurrence`;
* недопустимый переход, например закрытие регистрации до открытия, возвращает
  409/422 и не создаёт `Announcement`;
* ошибки отсутствующих стоек, выхода, языка, конфигурации или prompt возвращают
  понятную диагностику;
* покрыты application/API tests для нового read model, если он добавлен.

Frontend:

* `/dispatcher` открывает рабочий экран диспетчера;
* основные действия отображаются как кнопочные фильтры внутри одного экрана;
* переключение фильтра сразу перестраивает общий список подходящих рейсов;
* диспетчер может найти и выбрать рейс без перехода между страницами;
* недоступные рейсы скрыты из основного списка или показаны отдельно с причиной;
* языки выбираются из справочника, без ручного ввода кода;
* стойки и выход выбираются из активных справочников;
* запуск объявления создаёт `Announcement` через API;
* успешный запуск показывает подтверждение и созданный announcement id или
  краткий статус;
* ошибки 422/409 показываются рядом с формой;
* экран не содержит controls администрирования конфигураций.

```bash
php bin/phpunit
```

```bash
npm test
```

Manual QA:

```text
1. Войти как пользователь с доступом к dispatcher UI.
2. Открыть /dispatcher.
3. Нажать режим "Начало регистрации" и проверить, что отображаются рейсы без
   открытой регистрации.
4. Запустить объявление открытия регистрации с двумя стойками и одним языком.

   которого регистрация уже открыта.

   перехода статуса.
7. Проверить, что созданные действия создали `Announcement`.
8. Проверить, что недоступная конфигурация не запускается и показывает причину.
```
