# AeroFlow Architecture

## Цель документа

Этот документ описывает архитектуру платформы AeroFlow.

Документ является источником истины для разработки backend, frontend, playback-сервиса и audio-agent.

---

# Общая архитектура

AeroFlow состоит из четырёх основных репозиториев:

```text
aeroflow-core
aeroflow-web
aeroflow-playback
aeroflow-agent
```

Дополнительно существует общий репозиторий документации:

```text
aeroflow-docs
```

---

# Компоненты системы

## aeroflow-core

Основной backend-сервис.

Технологии:

* PHP
* Symfony
* PostgreSQL
* Symfony Messenger
* RabbitMQ

Ответственность:

* рейсы;
* расписание;
* статусы рейсов;
* объявления;
* шаблоны объявлений;
* аудиофайлы;
* пользователи;
* роли;
* аудит.

`aeroflow-core` содержит бизнес-логику аэропорта.

Он не воспроизводит аудио напрямую.

---

## aeroflow-web

Frontend-приложение.

Технологии:

* React
* Mantine
* TypeScript

Ответственность:

* интерфейс администратора;
* интерфейс диспетчера;
* управление рейсами;
* управление расписанием;
* управление объявлениями;
* просмотр очереди воспроизведения;
* управление аудиофайлами.

---

## aeroflow-playback

Сервис управления очередью воспроизведения.

Ответственность:

* принимать команды на воспроизведение;
* создавать задания воспроизведения;
* управлять очередью;
* учитывать приоритеты;
* запускать текущее объявление;
* останавливать текущее объявление;
* отменять будущие задания;
* обрабатывать экстренный режим;
* получать статусы от audio-agent.

`aeroflow-playback` не знает предметную область рейсов.

Он не должен содержать сущности `FlightDefinition`, `FlightSchedule`,
`FlightOccurrence`, `Gate`, `CheckInCounter`.

---

## aeroflow-agent

Локальный агент воспроизведения.

Ответственность:

* получать команды от `aeroflow-playback`;
* воспроизводить аудиофайлы;
* останавливать воспроизведение;
* ставить воспроизведение на паузу;
* продолжать воспроизведение;
* управлять громкостью;
* отправлять статусы выполнения.

`aeroflow-agent` не содержит бизнес-логику.

# Пример сценария

## Начало регистрации

1. Диспетчер выбирает `FlightOccurrence` в интерфейсе.
2. Диспетчер выбирает стойки регистрации.
3. Диспетчер нажимает "Открыть регистрацию".
4. `aeroflow-core` проверяет допустимость перехода occurrence (авторитет — агрегат).
5. `aeroflow-core` готовит `Announcement` по шаблону карточки рейса; готовность
   `AudioSequence` — предусловие: при неудаче переход отклоняется и ничего не
   сохраняется.
6. `aeroflow-core` атомарно (одна локальная транзакция) переводит occurrence в
   `check_in_open` и сохраняет `Announcement`.
7. После коммита контекст `Announcements` в `aeroflow-core` публикует команду
   `RequestAnnouncementPlayback` (асинхронно, RabbitMQ; durable outbox отложен —
   post-commit публикация из application layer).
8. `aeroflow-playback` создаёт `PlaybackJob` (идемпотентно по `messageId`).
9. Если очередь свободна, `aeroflow-playback` отправляет команду `PlayAudioSequence` агенту.
10. `aeroflow-agent` воспроизводит аудио.
11. После завершения агент отправляет статус выполнения.
12. `aeroflow-playback` отмечает задание как выполненное.
13. `aeroflow-core` получает событие о завершении при необходимости.

`FlightOccurrence` используется и для ручного диспетчерского запуска, и для
будущего запуска по расписанию. `FlightSchedule` остаётся возможным будущим
расширением.

# Коммуникация между сервисами

Основной способ взаимодействия — асинхронные сообщения.

Используется RabbitMQ.

## Core → Playback

Примеры команд:

* RequestAnnouncementPlayback
* CancelAnnouncementPlayback
* ActivateEmergencyMode
* DeactivateEmergencyMode

## Playback → Agent

Примеры команд:

* PlayAudioSequence
* StopCurrentPlayback
* PausePlayback
* ResumePlayback
* SetVolume

Транспорт зафиксирован: RabbitMQ в обе стороны (очереди `agent_commands` и
`agent_events`, дискриминаторы `command`/`event` в теле — ADR 002). Первый срез
(задача 016) реализует `PlayAudioSequence` и события started/completed/failed;
остальные команды — следующие срезы. Аудиофайлы агент получает по
download-контракту core (`GET /internal/v1/audio-assets/{id}/file`) и кэширует
локально по `audioAssetId`.

## Agent → Playback

Примеры событий:

* PlaybackStarted
* PlaybackCompleted
* PlaybackFailed
* PlaybackInterrupted

## Playback → Core

Примеры событий:

* AnnouncementPlaybackQueued
* AnnouncementPlaybackStarted
* AnnouncementPlaybackCompleted
* AnnouncementPlaybackCancelled
* EmergencyModeActivated
* EmergencyModeDeactivated

Обратный поток реализован для `Queued`, `Started` и `Completed`: playback
публикует их в RabbitMQ (очередь `playback_events`) после коммита своей локальной
транзакции, а `aeroflow-core` идемпотентно (по `messageId`) фиксирует факт приёма.
Статусы `Announcement` по этим событиям пока не меняются. Тело события несёт
дискриминатор `event`, поэтому получатель не зависит от PHP-классов издателя.

# Очередь воспроизведения

Все объявления превращаются в `PlaybackJob`.

`PlaybackJob` содержит:

* id;
* тип;
* приоритет;
* статус;
* время запуска;
* последовательность аудиофайлов;
* правило повтора;
* дату создания.

---

# Приоритеты

```text
1000  Экстренное объявление
100   Рейсовое объявление
50    Дополнительное объявление
0     Фоновая музыка
```

Правила:

* экстренное объявление прерывает текущую очередь;
* обычные объявления ставятся в очередь;
* фоновая музыка играет только когда очередь пуста;
* после завершения экстренного режима очередь продолжается.

## aeroflow-core

Хранит:

* расписание;
* объявления;
* шаблоны;
* аудиофайлы;

Не обязан иметь базу данных.

Может иметь локальный кэш файлов и конфигурацию.

# Совместимость со старой САО

## Подготовка составного объявления

`aeroflow-core` хранит редактируемый шаблон отдельно от факта запуска:

```text
FlightAnnouncementConfig
    -> AnnouncementVariant
        -> ordered template segments
```

Во время запуска Core получает snapshots выбранных стоек или выхода, разрешает
динамические слоты через Audio Catalog и сохраняет готовую неизменяемую
`AudioSequence` в `Announcement`.

```text
template + runtime parameters
    -> consumer-owned integration ports
    -> AudioSequence
    -> future RequestAnnouncementPlayback
```

Playback и Agent получают только готовую последовательность и не знают про
шаблоны, prompts, стойки или выходы.

Новая система должна поддерживать использование существующих аудиофайлов старой САО.

Старая структура:

```text
Звуки
├── Вылеты
├── Прибытие
├── Дополнительно
├── Экстренные
├── Числа
├── Гонги
└── Музыка
```

На первом этапе аудиофайлы могут использоваться как готовые assets.

В будущем возможна генерация объявлений через TTS.

# Режимы работы

## Ручной режим

Диспетчер вручную запускает объявления.

## Полуавтоматический режим

Система предлагает действия на основании расписания.

## Автоматический режим

Система самостоятельно создаёт объявления на основании расписания и статуса рейса.

# Архитектурные правила

1. `aeroflow-core` не должен напрямую воспроизводить звук.
2. `aeroflow-playback` не должен знать про рейсы и расписание.
3. `aeroflow-agent` не должен знать бизнес-логику.
4. Все важные действия должны логироваться.
5. Взаимодействие между сервисами должно быть асинхронным.
6. Команды должны быть идемпотентными.
7. Повторная доставка сообщения не должна ломать состояние системы.
8. Экстренный режим имеет максимальный приоритет.
9. Старые аудиофайлы должны поддерживаться как минимум на этапе миграции.
10. Документация в `aeroflow-docs` считается источником истины.
