# ADR 002: Дискриминация входящих сообщений между сервисами

Status: Accepted
Date: 2026-07-10

## Context

Интеграционные сообщения между `aeroflow-core` и `aeroflow-playback` пересекают
границу сервиса как нейтральный JSON: получатель не должен зависеть от PHP-классов
издателя (`type`-заголовок Messenger игнорируется при потреблении).

Сейчас направления асимметричны:

* **Core → Playback** (очередь `announcement_playback`): один тип сообщения —
  `RequestAnnouncementPlayback`. Сериализатор playback
  (`AnnouncementPlaybackSerializer`) безусловно строит эту команду из тела;
  дискриминатора типа в контракте нет.
* **Playback → Core** (очередь `playback_events`): три факта на одной очереди,
  различаются полем `event` в теле (`announcement_playback.queued|started|completed`).

По целевой модели playback будет принимать ещё `CancelAnnouncementPlayback`,
`ActivateEmergencyMode`, `DeactivateEmergencyMode`. Безусловный сериализатор
распарсит их как request и сломается. Нужно заранее зафиксировать, как расширяется
входящая сторона, чтобы срез cancel/emergency не решал это случайно внутри
serializer или message handler.

## Decision

Три уровня, по возрастанию стоимости изменения:

1. **Контракт (главное).** При появлении второго типа сообщения на одной очереди
   в тело добавляется нейтральный дискриминатор — поле `command` для команд
   (зеркально полю `event` исходящих событий), например
   `announcement_playback.request`, `announcement_playback.cancel`. Дискриминатор
   живёт в теле сообщения, не в транспортных заголовках и не в PHP-классах.
   Это контрактное изменение: издатель (core) и получатель (playback) меняются
   согласованно, `schemaVersion` инкрементируется.

2. **Транспорты.** «Фабрика сериализаторов» не пишется — точкой привязки
   сериализатора является транспорт Messenger. Emergency-команды при своём срезе
   получают **отдельную очередь/транспорт** (`emergency_commands`) со своим
   сериализатором: экстренная команда не должна стоять в брокере позади bulk-очереди
   обычных запросов. Это операционная изоляция приоритета, а не переиспользование
   кода.

3. **Код.** Пока типов на одной очереди 2–3 — один сериализатор с `match` по
   `command` и приватными гидраторами на тип. Registry декодеров
   (`supports()/decode()` + tagged iterator) не вводится заранее: потолок входящих
   команд playback — четыре, две из которых уходят на emergency-транспорт.
   Registry рассматривается только если типов на одной очереди станет ~5+ или
   валидация на тип станет нетривиальной.

## Consequences

* Текущий срез (015) не меняется: на `announcement_playback` один тип, безусловный
  сериализатор корректен.
* Срез `CancelAnnouncementPlayback` обязан: добавить `command` в контракт запроса
  и отмены, обновить издателя в core, заменить безусловное построение в
  `AnnouncementPlaybackSerializer` на `match` с ошибкой на неизвестный `command`.
* Срез emergency mode обязан завести отдельный транспорт, а не подмешивать команды
  в общую очередь.
* Исходящая сторона уже соответствует этому решению (поле `event`) и не меняется.
