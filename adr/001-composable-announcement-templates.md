# ADR 001: Composable Announcement Templates

Status: Accepted  
Date: 2026-06-23

## Context

Текущий `FlightAnnouncementConfig` хранит для каждого языка один
`AnnouncementVariant`, источником которого является целый `AudioAsset` или текст.
Этого недостаточно для рейсовых объявлений, которые должны собираться из
нескольких частей:

```text
гонг
-> записанная фраза
-> выбранные стойки регистрации
-> записанная фраза
```

Стойки регистрации и выход на посадку выбираются диспетчером во время запуска.
Их значения нельзя заранее зашить в один аудиофайл. Playback и Agent при этом не
должны знать семантику рейса, стоек, выхода или шаблона.

Legacy САО использовала отдельные числовые WAV-файлы, однако имя файла и
физический путь не должны становиться бизнес-контрактом AeroFlow.

## Decision

### Template model

Существующий `FlightAnnouncementConfig` развивается, а параллельный агрегат
`AnnouncementTemplate` не вводится.

Каждый языковой `AnnouncementVariant` содержит упорядоченную коллекцию сегментов:

* `audio_asset` — ссылка на готовый `AudioAsset`;
* `dynamic_slot` — типизированное значение, разрешаемое во время запуска;
* `pause` — явная пауза заданной длительности;
* `text` — утверждённый текст для будущего TTS.

Первая версия поддерживает динамические слоты:

* `check_in_counters`;
* `gate_code`.

`TextSegment` может храниться и редактироваться, но без TTS конфигурация с таким
сегментом не считается готовой к воспроизведению.

### Runtime model

Модель разделяется на три представления:

```text
FlightAnnouncementConfig
    -> редактируемый шаблон

Announcement
    -> факт запуска и snapshot операционных параметров

AudioSequence
    -> неизменяемый результат разрешения шаблона
```

При создании `Announcement` Core:

1. проверяет выбранный рейс и операционные параметры;
2. сохраняет ID и коды выбранных стоек или выхода как snapshot;
3. выбирает активные языковые варианты;
4. разрешает каждый сегмент в упорядоченную последовательность `AudioAsset` ID;
5. сохраняет готовую `AudioSequence`.

Если обязательный сегмент нельзя разрешить, объявление не считается
подготовленным и не передаётся в Playback.

Playback получает только готовую последовательность. Он не получает шаблон,
динамические слоты, рейс, стойки или выход.

### Bounded contexts

`Flight Operations` владеет справочниками:

* `CheckInCounter`;
* `Gate`.

Они имеют UUID, уникальный строковый код, отображаемое имя, порядок отображения
и признак активности. Коды могут быть числовыми или составными: `1`, `12`, `A5`,
`B-2`.

`Audio Catalog` владеет `AudioPrompt` — семантической связью между значением и
физическим аудиофайлом:

```text
kind + value + languageCode -> AudioAsset
```

Первая версия поддерживает виды:

* `check_in_counter_code`;
* `gate_code`.

Имя и путь `AudioAsset` не используются для поиска. Составные коды не
декомпозируются автоматически: для `A5` требуется точный активный prompt.

`Announcements` взаимодействует с обоими контекстами только через
consumer-owned application ports и адаптеры в `Infrastructure/Integration`.

### API transition

Публичный контракт запуска объявления меняется без переходного периода:

```text
checkInCounterStart/checkInCounterEnd -> checkInCounterIds[]
gateCode                              -> gateId
```

Созданный `Announcement` сохраняет как идентификаторы, так и коды, чтобы
последующие изменения справочника не меняли уже подготовленное объявление.

### Migration

Существующие `AnnouncementVariant` автоматически преобразуются:

* `audio_asset` — в вариант с одним `AudioAssetSegment`;
* `text` — в вариант с одним `TextSegment`.

Данные конфигураций не сбрасываются, параллельная legacy-модель не сохраняется.

## Consequences

Положительные:

* шаблоны становятся составными и расширяемыми;
* аудиофрагменты переиспользуются;
* динамические значения фиксируются на момент запуска;
* Playback и Agent остаются изолированными от аэропортовой модели;
* legacy-аудиотека может импортироваться через семантические prompts.

Отрицательные:

* усложняются модель конфигурации, API и миграция;
* готовность шаблона требует межконтекстной проверки;
* каждый язык должен иметь полный набор необходимых prompts;
* изменение API запуска требует одновременного обновления клиента.

## Out of Scope

* TTS;
* языковые fallback-цепочки;
* автоматическая композиция букв и чисел;
* текущее и плановое время;
* номер рейса и аэропорты как dynamic slots;
* передача последовательности в `aeroflow-playback`.
